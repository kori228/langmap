# LangMap セグメント検証・修正プロンプト

## 概要

LangMap の各言語のセグメント設定を検証・修正するためのプロンプト集。
過去の検証で発見された問題パターンと、その対処法を網羅。

---

## 1. 検証プロンプト（全件チェック用）

### 使用方法
- 言語グループごとに分けて並行実行（4グループ推奨）
- 同一プロンプトで複数回実行し、クロスバリデーションで一致した問題のみ修正

### プロンプト本文

```
data.js を読み込み、指定された言語グループの全100文について、セグメントレベルの検証を行ってください。

■ 参照言語の決定ルール
- 日本語方言 (ja_*) → ja
- 韓国語方言 (ko_*) → ko
- 中国語方言 (zh_*, yue, nan, wuu, hak_cn, cdo) → zh
- 英語方言 (en_*) → en
- スペイン語方言 (es_*) → es_eu
- フランス語方言 (fr_*) → fr
- ドイツ語方言 (de_*, nds) → de
- アラビア語方言 (ar_*) → ar
- イタリア語方言 (nap, scn, vec) → it
- ポルトガル語 (pt_br) → pt_eu
- タイ語方言 (th_*) → th
- ノルウェー語ニーノシュク (nn) → no
- 内モンゴル語 (mn_cn) → mn
- タジク語 (tg) → fa
- クルド語ソラニー (ckb) → ku
- その他すべて → en

■ チェック項目（6種類）

1. 【スワップ】セグメントIDの入れ替え
   - 言語XのセグメントAのテキストが、参照言語のセグメントBの意味に対応している
   - 例: s25-ar_sd: [C]"شركة"(company) は参照の D=company に対応。C と D が入れ替わっている

2. 【結合】複数セグメントの不正な結合
   - 1つのセグメントに、参照言語の2つ以上のセグメントの意味が含まれている
   - 例: s29-ilo: [C]"nga agtokar iti piano" に C=play と B=piano が結合

   ★★ 検出方法（重要）★★
   各セグメントのテキストを単語単位で分解し、参照言語の各セグメントの
   意味と照合する。1つのセグメント内に、参照言語の異なるセグメントIDに
   対応する単語が混在していないかチェックする。

   具体的な手順:
   a. 参照言語の各セグメントの意味を把握（例: en B="the piano", C="play"）
   b. 対象言語の各セグメントテキストを読み、その中の各単語/フレーズが
      参照のどのセグメントIDに対応するか判定
   c. 1つのセグメント内に2つ以上の参照セグメントIDに対応する内容があれば
      【結合】として報告

   ★ IDスワップだけでなく、必ずこの結合チェックも行うこと。
   ★ 「IDを入れ替えるだけ」では修正できない問題がこのタイプ。
   ★ 修正にはテキストの分割（単語境界での切断）が必要。

   - 分割すべき典型パターン:
     - 動詞+目的語: 「弾くピアノを」→ [C]弾く [B]ピアノを
     - 場所格助詞+場所: 「公園で」→ [E]公園 [B]で
     - 前置詞+名詞: 「in the park」→ [B]in [E]the park
     - 属格助詞: 「醫館之畔」→ [C]醫館之 [B]畔
     - 受身マーカー+動詞: 「被雪覆盖」→ [C]被 [B]雪 [C]覆盖
     - 疑問助詞: 「좋다고」→ [C]좋다 [D]고
     - 動詞ペア: 「在...焉」→ [D]在...[D]焉、「欲...焉」→ [E]欲...[E]焉
     - 動詞+前置詞+目的語: 「agtokar iti piano」→ [C]agtokar [B]iti piano

3. 【欠落】参照言語にあるセグメントIDが存在しない
   - 80%以上の言語が持つセグメントIDを、この言語が持っていない場合に報告
   - ただし、主語省略(pro-drop)言語でA(I/私)がないのは許容

4. 【余分】参照言語にないセグメントIDが存在する
   - 関係代名詞(G)の追加など、言語構造上必要な場合は許容

5. 【未定義】文のsegments定義に存在しないIDを使用
   - 必ず修正が必要

6. 【隣接同一ID】連続する2つのセグメントが同じID
   - 必ずマージが必要（テキストをスペースで結合）

■ 報告形式

問題があるもののみ報告。形式:
s{番号}-{言語コード}: 【{種類}】{説明}

例:
s25-ar_sd: 【スワップ】C="شركة"(company)はD、D="كبيرة"(big)はC
s29-ilo: 【結合】C="nga agtokar iti piano" にC=play(agtokar)とB=piano(iti piano)が結合
s20-pl: 【欠落】D(rain noun)が欠落、B="pada deszcz"にBとDが結合
s58-egy: 【未定義】セグメントDを使用しているが、この文のsegmentsにはA,B,Cのみ定義
```

## 2. 修正プロンプト（問題修正用）

### プロンプト本文

```
data.js の以下のセグメント問題を修正してください。

■ 修正ルール（厳守）

1. テキスト内容は変更しない — セグメントIDの付け替えと境界の分割のみ
2. 隣接する同一IDのセグメントを作らない — 発生する場合はテキストをスペースで結合
3. すべてのセグメントIDは文のsegments定義に存在するものを使用
4. 参照言語のセグメント意味に合わせてIDを割り当てる
5. 分割時は単語境界で切る — 形態素の途中で切らない

■ 修正パターン別の手順

★★ 重要: スワップと結合は同時に修正すること ★★
スワップのみ修正して結合を放置すると、後続の修正エージェントが
スワップ修正時に結合状態を再導入する恐れがある。
1つの文×1つの言語に対して、スワップ・結合・欠落をすべて一度に修正する。

【スワップ修正】
- テキストはそのまま、IDだけ入れ替え
- 2-way: A↔B → IDを交換
- 3-way: A→B, B→C, C→A → 同時に付け替え
- 修正前後の語順は変えない

【結合修正】
- 結合されたテキストを単語境界で分割し、それぞれに正しいIDを付与
- 例: [C,"nga agtokar iti piano"] → [C,"nga agtokar"], [B,"iti piano"]
- 分割位置の判断基準: 参照言語のセグメント境界に対応する位置
- ★ スワップ修正と同じパスで実行すること（別パスにすると戻る可能性あり）

【欠落修正】
- 既存セグメントのテキストから該当部分を分離して新セグメントを作成
- 例: [B,"公園で"] → [E,"公園"], [B,"で"]
- ★ これも結合の一種。スワップと同時に修正すること

【未定義修正】
- 最も意味的に近い定義済みIDに変更
- 隣接同一IDが発生する場合はマージ

■ 修正結果の記録

各修正について以下を記録:
## s{番号}-{言語}: {問題の説明}
- BEFORE: {修正前のセグメント配列}
- AFTER: {修正後のセグメント配列}
```

### A. 語順に起因するスワップ（体系的）

| 言語 | パターン | 原因 |

| ar_sd, ar_tn | 広範囲のA-E全面入れ替え | 方言の語順がMSAと大幅に異なる |

### E. 歴史言語の注意点

| 問題 | 対応 |
|---|---|
| 現代借用語 | Wi-Fi, AI, sushi, smartphone等は迂言に置換 |
| 焉の誤用 | 名詞末尾に付けない。文末助詞として正しい位置に |
| 之の位置 | 修飾される名詞の前に付く（醫館之 [B]畔） |
| 旬 ≠ 週 | 旬=10日、週=七日 |
| 時辰 ≠ 時間 | 一時辰=2時間、半時辰=1時間 |
| 두음법칙 | 近世韓国語は漢字語のみ適用（려행、류학）、固有語は不適用（여기、여름） |
| 機械/기계 | 中世では「얼개」等の固有語を使用 |

### F. s81（頭痛）の特殊ケース

この文は6セグメント(A,B,C,D,E,F)を持ち、2つの表現パターンがある：
- E/Fパターン: E=headache(名詞), F=have → 「頭痛がある」型（ko, en）
- B/Dパターン: B=head, D=hurts → 「頭が痛い」型（ja, ko_kp, zh, ru）

方言は親言語と同じパターンを使用すべき。

## 4. クロスバリデーション手順

1. 同一プロンプトで最低3回実行（異なるエージェント）
2. 各回の結果をキー（s番号-言語コード）で正規化
3. 2回以上で一致した問題のみ修正対象とする
4. 修正後、再度検証を1回実行して残存問題を確認
5. Adjacent=0, Invalid=0, Empty=0 を確認して完了

## 6. 小グループ内一致検証（異常検出手法）

### 概要

類似した言語・方言同士は、セグメントIDの並びがほぼ一致するはず。
この前提を利用して、小グループ内でセグメントID列を比較し、逸脱を検出する。

エージェントの言語知識に依存しないため、プログラム的に実行でき、偽陽性が少ない。

**注意: これは異常検出（スクリーニング）手法であり、最終判断ではない。**
検出された逸脱は「要確認」のフラグであり、実際に問題かどうかはセクション1の検証プロンプトまたは人間の判断で確定する。言語構造上の正当な差異（語順の違い、pro-drop等）も検出されるため、すべてが修正対象ではない。

### 原理

```
日本語標準: [A]私は [B]公園で [C]遊んで [D]いる → ID列: A-B-C-D
大阪弁:     [A]うちは [B]公園で [C]遊んで [D]んねん → ID列: A-B-C-D ✓ 一致
博多弁:     [A]うちは [C]公園で [B]遊んで [D]ばい → ID列: A-C-B-D ✗ B/C逆転！
```

同じ語順の言語同士なら、ID列は一致すべき。不一致は問題の可能性が高い。

### 小グループ定義

各グループ内の言語はセグメントID列がほぼ一致することを期待。
代表言語（★）を基準にし、他のメンバーの逸脱を検出。


### 検証プロンプト（プログラム的）

```javascript
// 小グループ内でのID列一致検証スクリプト
const groups = {
  'ja方言':     { ref: 'ja', members: ['ja_kyo','ja_osa','ja_hir','ja_hak','ja_aom','ja_oki','ja_edo','ja_heian'] },
  'ko方言':     { ref: 'ko', members: ['ko_kp','ko_bus','ko_jeju','ko_yb','ko_mid','ko_em'] },
  'zh普通話系': { ref: 'zh', members: ['zh_db','zh_sc'] },
  'zh南方系':   { ref: 'zh', members: ['yue','nan','wuu','hak_cn','cdo'] },
  'en方言':     { ref: 'en', members: ['en_aave','en_south','en_app','en_au','en_in','en_ie','en_sco','en_yk','en_ck','en_sg'] },
  'es方言':     { ref: 'es_eu', members: ['es_mx','es_ar','es_co','es_cl','es_cu','es_pe','es_an'] },
  'fr方言':     { ref: 'fr', members: ['fr_qc','fr_af','fr_be'] },
  'de方言':     { ref: 'de', members: ['de_gsw','de_at','de_by'] },
  'ar方言':     { ref: 'ar', members: ['ar_eg','ar_lev','ar_gulf','ar_iq','ar_ma','ar_tn','ar_sd'] },
  'it系':       { ref: 'it', members: ['nap','scn','vec'] },
  'pt系':       { ref: 'pt_eu', members: ['pt_br'] },
  'th方言':     { ref: 'th', members: ['th_isan','th_n','th_s'] },
  'sv-no-da':   { ref: 'sv', members: ['no','da'] },
  'ru系':       { ref: 'ru', members: ['uk','be'] },
  'pl系':       { ref: 'pl', members: ['cs','sk'] },
  'hi系':       { ref: 'hi', members: ['ur','pa','bho'] },
  'id-ms':      { ref: 'id', members: ['ms'] },
  'tl系':       { ref: 'tl', members: ['ceb','ilo'] },
};

for (const [name, {ref, members}] of Object.entries(groups)) {
  for (let i = 0; i < 100; i++) {
    const refIds = S[i].langs[ref]?.map(x => x[0]).join('-') || '';
    members.forEach(m => {
      const mIds = S[i].langs[m]?.map(x => x[0]).join('-') || '';
      if (refIds && mIds && refIds !== mIds) {
        console.log(`s${i}-${m}: ID列 ${mIds} ≠ 参照${ref} ${refIds}`);
      }
    });
  }
}
```

### 逸脱の解釈

| 逸脱タイプ | 例 | 判断 |
|---|---|---|
| 完全一致 | A-B-C-D = A-B-C-D | ✓ 問題なし |
| 順序入れ替え | A-B-C-D vs A-C-B-D | ▲ 語順差？スワップ？要確認 |
| セグメント増減 | A-B-C-D vs A-B-C | ▲ 欠落？結合？要確認 |
| 完全不一致 | A-B-C-D vs D-C-A-B | ✗ 高確率で問題あり |

### 方言間不一致の許容基準

- **語順の違い**: 同一言語の方言間で語順が異なることは稀。不一致は問題の可能性95%以上
- **セグメント数の違い**: 方言で主語省略(pro-drop)する場合は許容。それ以外は結合問題の可能性大
- **セグメントの追加**: 関係代名詞や接続助詞の追加は許容（例: G セグメント）
- **歴史言語**: 語彙は異なるがID列は親言語と一致すべき（古い語順でも意味的対応は同じ）

### 実行手順

1. 上記スクリプトを実行し、小グループ内の逸脱をすべてリストアップ
2. 各逸脱を「語順差（許容）」「スワップ（修正）」「結合（分割）」に分類
3. 修正が必要なもののみ対応
4. 修正後、スクリプトを再実行して逸脱数が減少したことを確認
